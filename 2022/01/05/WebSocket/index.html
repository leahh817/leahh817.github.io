WebSocket

WebSocket是一种在单个TCP连接上提供全双工通信通道的通信协议。这个定义感觉听过千万遍，但是对WebSocket的理解还只停留在“即时通信”上，知其然不知其所以然。下面想列一下几种其他即时通信方式来进行一下比较，看一下全双工通信相对于其他单工和半双工的优势。

  • （短）轮询（Polling）
    顾名思义，client会隔一段时间就向sever发送http请求，服务器收到请求后把最新的数据返回。但是client端不知道什么时候sever中的数据会有更新，因此它需要定时发请求询问，但是如果这段时间内sever一直没有更新过，那这些请求实际上就并没有什么意义，反而浪费带宽。

![image-20211221183553356](/Users/wenxin.chen/Library/Application Support/typora-user-images/image-20211221183553356.png)

显然轮询的方式下要想获取最新数据必须先发送请求，但对实时性要求不一致的地方用固定的频率难免会造成浪费带宽或者不那么实时的局面，而且就实现方式而言，需要循环判断是否达到设置的time值这本身也需要消耗一定的资源。而下面的Comet则是基于http的长连接的，会在client和sever之间保持一个长连接，数据发生变化时，sever主动将最新数据推送给cilent，其中Comet细分为两种，当然这两种长连接的方式都避不开的缺点是保持长连接所带来的开销。

  • Comet

    • 长轮询（Long-Polling）

      如图所示，client发送请求之后，sever并不立即返回，而是在有数据更新后才响应。如果没有更新则会一直将请求保持，等待有更新事件发生。但是如果很长时间都没有更新，一段时间后该请求就会超时，client在知道超时后就会再重新发送一个请求。

    ![image-20211221184137323](/Users/wenxin.chen/Library/Application Support/typora-user-images/image-20211221184137323.png)

    但是其实可以看到长轮询本质上还是一种轮询，想要获取最新的数据也必须发送请求。虽然server会等待更新完后才返回，看似相比短轮询节约了一些无效的轮询请求，但是也有它的弊端。在sever响应client之后，client再次发送请求到sever的这个往返时间差在一些更新频率较快的情况下就无法忽略了。

    • 流技术（Http Streaming）

    流技术是基于html的ifream的，它的src属性会保持制定sever的长连接请求，这样sever就可以一直不断的向client返回数据。如图，建立连接后，只要数据发生变化sever就可以给client发送请求，链接不会断开。但是这样似乎对有强迫症的冲浪选手不太友好，因为你的页面会一直处于加载状态。

    ![image-20211221184303767](/Users/wenxin.chen/Library/Application Support/typora-user-images/image-20211221184303767.png)

  • SSE（Sever-Sent Events）

SSE的原理与长轮询机制的流技术类似，连接建立后sever可以一直发送数据，直到某一方断开连接。但不同的是它服务器返回的响应内容为事件流（content-type —— text/event-stream），并且能在断开连接后自动重连。

  • WebSocket

前面介绍的都是基于HTTP协议的通讯方式，而WebSocket自成一派，自己就是一个协议。HTTP和WebSocket都是OSI模型的最高层也就是应用层的协议，它们都是基于TCP来传输数据的，或许可以简单一点理解为它们都是对TCP的封装。那WebSocket和HTTP有啥关系呢？对于 WebSocket 来说，它必须依赖HTTP协议进行一次握手，握手成功后，数据就直接从 TCP 通道传输，与 HTTP 无关了。

![image-20211221184904071](/Users/wenxin.chen/Library/Application Support/typora-user-images/image-20211221184904071.png)

如图可以看到，相对于前面基于http的几种方式，WebSocket是长连接模式并且提供的是一种全双工的通讯，WebSocket连接建立之后除非一方断开，否则client不需要每次都发送请求，client和sever之间进行的是双向通信。

从性能方面考虑，WebSocket>长连接SSE>长轮询>短轮询，从兼容性考虑则反过来。

![image-20211222100150035](/Users/wenxin.chen/Library/Application Support/typora-user-images/image-20211222100150035.png)

在 C 端创建一个 socket 实例，并为这个实例绑定其所要连接的 B 端的 IP 地址和端口号，在 B 端创建另一个 socket 实例，并且绑定本地端口号来进行监听。当 C和 B 建立连接后，双方就建立了一个端对端的 TCP 连接，从而可以进行双向通信。WebSocket是HTML5规范中的一部分，借鉴来socket的思想，为client和sever提供了类似的双向通信机制。Socket.IO是一个封装了 Websocket、基于 Node 的 JavaScript 框架，包含 client 的 JavaScript 和 server 的 Node，屏蔽了所有底层细节，让顶层调用更简单。而且Socket.IO不仅支持 WebSocket，还支持许多种轮询机制以及其他实时通信方式，并且封装了通用的接口。

比较总结

相比MQTT的小型消息，RESTful http在数据传输上更灵活，并且可以传输多种类型的信息如文件、对象甚至视频等等。因此对较大数据的传输，如检索历史数据、创建、读取、或需要对GET或POST请求作出响应的时候使用REST更加合适。

MQTT最初是为远程传感或设备控制等设备而设计的，为环境不太稳定的设备提供可靠的传输服务。因此MQTT在功率受限、小型设备和远程设备上优势较高，并且它可以提供可靠、一对多的传输服务。因此在设备耗电量有限，连接是断断续续的，带宽使用有限，希望立即将更新发布而不需要轮询服务器来获取新消息，以及需要一对多同时相互通信的协作场景下MQTT更加适合。

简单来说，http主要用于服务端向客户端提供消息，或客户端主动去服务器请求http接口调取数据的情形，它并没有主动给客户端推送消息功能。而mqtt主要是用于移动端主动向服务端推送消息，并没有去请求消息的功能。

前面说到WebSocket和HTTP都是应用层的协议，但是WebSocket提供了它的 “竞争对手” RESTful http在同一级别上没有提供的东西——基于TCP的单个连接，允许低延迟的双向通信，而不需要在每次请求时打开和关闭。MQTT也是应用层协议，它和WebSocket都基于TCP连接并且都支持双向的通信。MQTT是为物联网场景设计的基于TCP的Pub/Sub协议，而WebSocket是为了HTML5应用方便与服务器双向通讯而设计的协议。两者的交集在于如果要通过HTML5应用来作为MQTT的客户端，以便接受设备消息或者向设备发送信息，那么MQTT over WebSocket自然成了最合理的途径。也就是说MQTT可以工作在WebSocket之上，使用全双工低时延的WebSocket来可以帮助MQTT的特性和优势的发挥。
